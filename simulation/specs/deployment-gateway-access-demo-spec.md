# Deployment And Gateway Access Demo Spec

## 1. Purpose

This real-platform demo shows the core platform value: a Docker image can be deployed as a Kubernetes microservice, but visitors only access it through the centralized gateway.

The key expo story is not just `we deployed a service`. The stronger message is:

`The service stays internal, and the gateway becomes the safe public front door.`

---

## 2. Existing Platform Feature Being Shown

- Dashboard `Deploy New Service` workflow
- Service inventory list on the main dashboard
- The generated `public_url`
- Service details page with the visible public link
- The real gateway route shape: `http://localhost:30080/<service-name>`

---

## 3. Sample Service Or Setup Needed

### Recommended service

Use `example-api.tar` from `examples/simple-api`.

### Deployment values

- `Service Name`: `example-api`
- `Container Port`: `8000`

### Why this service is best

- it is simple and stable
- it responds quickly
- it gives a clean success story without distracting security issues
- it is enough to prove gateway-only public access

---

## 4. Presenter Goal

In 30 to 60 seconds, show that:

1. the operator can deploy a user service from the dashboard
2. the platform gives a working public route
3. that route is the gateway route, not direct pod access
4. the architecture keeps user services internal while still making them usable

---

## 5. Exact Presenter Steps

1. Open the dashboard at `http://localhost:30000`.
2. Go to the `Deploy New Service` card.
3. Upload `example-api.tar`.
4. Fill:
   - `Service Name`: `example-api`
   - `Container Port`: `8000`
5. Click the deploy button.
6. When the success box appears, point to:
   - `Success! Service is live.`
   - the `Visit http://localhost:30080/example-api` link
7. Open the public link.
8. Show that the service responds through the gateway URL.
9. Return to the dashboard and point at the service row in the table.
10. Click `Manage`.
11. On the service details page, point to the public URL again.

### Optional stronger proof step

If the presenter is comfortable using a terminal during the expo:

1. Say that the service itself is internal in Kubernetes.
2. Do not browse directly to a pod IP or internal ClusterIP.
3. Point out verbally that the public link always follows the gateway route pattern.

This keeps the demo clean without relying on risky live infrastructure probing.

---

## 6. What The Visitor Should See

- a normal upload-and-deploy flow
- a success state instead of code output
- a visible public link generated by the platform
- a working page at the gateway URL
- a service details page that keeps the same route visible

---

## 7. Message To Explain

Use this layman explanation:

`The deployed service is like an office inside the building. Visitors do not go directly to that office door. They first pass through the main gate, and that gate decides how traffic should enter.`

Shorter version:

`The service stays inside. The gateway is the only public door.`

---

## 8. Proof To Point Out On Screen

Point to these exact proof items:

- the `Deploy New Service` card
- the success state after deployment
- the generated `Visit ...` link
- the public URL format `http://localhost:30080/<service-name>`
- the same public URL on the service details page

Best proof sentence:

`The proof is that the platform publishes one clean gateway URL for the service instead of exposing the service directly.`

---

## 9. Setup Instructions

Before the expo:

1. Build and save `example-api.tar`.
2. Start the platform and verify the dashboard opens at port `30000`.
3. Make sure the gateway responds at port `30080`.
4. Run one full deployment test with `example-api` before visitors arrive.
5. Keep the tar file on the desktop or in another easy-to-reach location.

---

## 10. Reset Instructions

Between visitors:

1. If the service is already deployed and working, reuse it for the next run to save time.
2. If you want the full deploy animation again, delete the service and redeploy it.
3. Keep the service name predictable, preferably `example-api`, so the public URL stays easy to say aloud.
4. If the dashboard becomes cluttered, remove non-demo services before the next round.

---

## 11. Practical Expo Notes

- This is the best foundation demo to run before Nikto or WAF explanations.
- Keep the focus on architecture and controlled access, not on Kubernetes jargon.
- Avoid spending time on internal service networking details unless judges specifically ask.
