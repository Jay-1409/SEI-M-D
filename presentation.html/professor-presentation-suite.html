Slide 2 · Problem

Problem and Threat Model

Problem:
  • Microservices often get exposed directly after deployment.
  • Security controls become inconsistent across teams.
  • Observability is weak when enforcement is scattered.
Threats addressed:
  • SQL injection and XSS payloads
  • Abusive/flood traffic
  • Unauthorized access to sensitive routes
Phrase: "safe access, not zero access".

Slide 5 · Controls

Gateway Security Controls

Traffic protection:
  • WAF checks for SQLi and XSS patterns
  • Header sanitization and content-type enforcement
  • Structured block event logging
Access and abuse control:
  • Per-service and per-route rate limits
  • Route-level API key protection
  • Redis-backed policy state and counters
Mention 403, 429, 401 here once as anchor outcomes.
Slide 6 · Scanning

Two-Layer Vulnerability Validation

Trivy (image layer):
  • Scans container image contents
  • Reports severity counts
  • Good for pre-trust package risk
Nikto (runtime layer):
  • Scans running web endpoint behavior
  • Finds exposed server/web weaknesses
  • Complements image scanning
Key phrase: "different layers, different answers".

Slide 8 · Limits

Limitations and Honest Tradeoffs

  • Regex-based detection can have false positives/false negatives.
  • Current setup depends on Redis availability for richer state.
  • Production hardening needs stricter policy gates.
Why this is still strong:

It is a practical, demonstrable foundation where security is built into platform flow, not added as an afterthought.

Professors appreciate clear limits when paired with rationale.
